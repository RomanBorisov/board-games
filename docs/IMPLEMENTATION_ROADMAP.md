# Детальный roadmap реализации (следующие шаги)

Этот документ фиксирует пошаговый план разработки после инициализации монорепозитория. План ориентирован на MVP игры "Дурак" и расширение платформы под другие настольные игры.

## 1) Цели и рамки roadmap

- Довести проект до состояния production-ready MVP: авторизация, лобби, игровая сессия, real-time матч.
- Сохранить архитектурный принцип: транспортный слой не зависит от правил конкретной игры.
- Соблюдать принятые стандарты:
  - Angular состояние через Signal API (`signal`, `computed`, `effect`).
  - CSS/SCSS по БЭМ.
- Заложить базу для масштабирования: новые игры добавляются через новый engine, без ломки существующей логики.

## 2) Общая последовательность фаз

1. Фундамент контрактов и доменной модели.
2. Данные и инфраструктура (Prisma, PostgreSQL, Redis, Kafka, Docker).
3. Авторизация и безопасность (REST + WS).
4. Сессии и лобби.
5. Игровой движок "Дурак".
6. WebSocket-геймплей.
7. Frontend игровые экраны и UX.
8. Качество, тесты, наблюдаемость.
9. Production-подготовка и релиз.

## 3) Фаза A — Домен и shared-контракты (1-2 дня)

### Задачи

- Зафиксировать общие типы в `shared`:
  - `Card`, `Suit`, `Rank`.
  - `Player`, `GameType`, `SessionStatus`.
  - контракты WS-событий (клиент -> сервер, сервер -> клиент).
- Зафиксировать интерфейс `GameEngine`:
  - `initGame`.
  - `processAction`.
  - `getStateForPlayer`.
  - `isGameOver`.
  - `getWinner`.
- Определить DTO для REST/WS:
  - Auth DTO.
  - Session DTO.
  - Game action DTO.

### Критерий готовности

- Frontend и backend импортируют единые типы из `@board-games/shared`.
- Нет дублирующихся локальных интерфейсов для одних и тех же сущностей.

## 4) Фаза B — Данные и инфраструктура (2-3 дня)

### Задачи по БД и ORM

- Подключить Prisma в backend.
- Описать `schema.prisma`:
  - `User`.
  - `GameSession`.
  - `GameLog`.
  - enum-ы статусов и типов игр.
- Создать и применить первую миграцию.
- Добавить seed-скрипт (опционально, для локальной разработки).

### Задачи по Redis

- Спроектировать ключи:
  - `session:{id}:state`.
  - `session:{id}:players`.
  - `jwt:blacklist:{jti}`.
- Добавить service-слой для чтения/записи состояния игры.
- Ввести TTL там, где данные должны "протухать" автоматически.

### Задачи по Kafka

- Создать базовые топики:
  - `game-events`.
  - `session-events`.
- Реализовать producer (публикация событий).
- Реализовать consumer (асинхронная запись логов/статусов).

### Задачи по окружению

- Подготовить `docker-compose.dev.yml` для:
  - PostgreSQL.
  - Redis.
  - Zookeeper + Kafka.
  - Kafdrop.
- Подготовить `.env.example` со всеми обязательными переменными.

### Критерий готовности

- Локально поднимаются все инфраструктурные сервисы.
- Backend успешно подключается к PostgreSQL/Redis/Kafka.
- Есть миграция и рабочая схема данных.

## 5) Фаза C — Авторизация и безопасность (3-4 дня)

### Backend

- Реализовать `UsersModule`:
  - создание пользователя.
  - получение по username/id.
- Реализовать `AuthModule`:
  - `POST /api/auth/register`.
  - `POST /api/auth/login`.
  - `POST /api/auth/refresh`.
  - `POST /api/auth/logout`.
  - `GET /api/auth/me`.
- Внедрить bcrypt для хеширования паролей.
- Реализовать JWT:
  - access token.
  - refresh token.
  - `jti` для blacklist.
- Добавить guards:
  - HTTP guard для защищенных REST-ручек.
  - WS guard для handshake сокета.
- Включить rate limit для auth endpoint-ов.

### Frontend

- Создать экраны:
  - `/auth/login`.
  - `/auth/register`.
- Реализовать auth state через Signal API:
  - текущий пользователь.
  - access token.
  - состояние загрузки.
  - состояние ошибок.
- Добавить HTTP interceptor:
  - подстановка `Authorization: Bearer ...`.
  - обработка 401 + сценарий refresh.
- Добавить route guard для приватных роутов.
- Реализовать "гостевой вход":
  - сохранение guest-name для сессий.

### Критерий готовности

- Пользователь может зарегистрироваться, войти, обновить токен, выйти.
- Защищенные endpoint-ы/роуты недоступны без валидного токена.
- WS-подключение отклоняется при невалидном токене.

## 6) Фаза D — Сессии и лобби (2-3 дня)

### Backend

- Реализовать `SessionsModule`:
  - `POST /api/sessions` (создание).
  - `GET /api/sessions/:id` (получение).
- При создании сессии:
  - генерировать `sessionId`.
  - сохранять создателя (user или guest).
  - публиковать `SESSION_CREATED` в Kafka.
- Реализовать проверку жизненного цикла сессии:
  - `WAITING`.
  - `IN_PROGRESS`.
  - `FINISHED`.
  - `ABANDONED`.

### Frontend

- Реализовать страницу лобби:
  - кнопка "Создать игру".
  - отображение join-ссылки.
  - кнопка копирования ссылки.
- Реализовать страницу входа в сессию:
  - `/game/:sessionId/join`.
  - поведение для auth-пользователя и для гостя.

### Критерий готовности

- Игрок создает сессию и получает рабочую ссылку приглашения.
- Второй игрок может присоединиться по ссылке.

## 7) Фаза E — Реализация DurakEngine (3-5 дней)

### Задачи

- Реализовать `DurakState`:
  - колода.
  - козырь.
  - руки игроков.
  - стол (пары attack/defense).
  - очередь хода.
  - фаза хода.
- Реализовать правила:
  - выбор атакующего.
  - атака.
  - подбрасывание.
  - защита.
  - взятие.
  - пас/завершение хода.
  - добор до 6.
  - определение конца игры и победителя.
- Реализовать `getStateForPlayer`:
  - текущему игроку отправляется `myHand`.
  - сопернику отправляется только `cardCount`.

### Unit-тесты движка

- Тесты happy-path для полного раунда.
- Тесты валидации невалидных действий.
- Тесты edge-cases:
  - пустая колода.
  - попытка защититься неверной картой.
  - попытка атаковать не в свою очередь.
  - завершение игры при разных раскладах.

### Критерий готовности

- Все ключевые правила "Дурака" реализованы.
- Тесты движка стабильно проходят.

## 8) Фаза F — WebSocket игровой контур (4-5 дней)

### Backend

- Реализовать `GameGateway` (Socket.io):
  - `join_session`.
  - `attack`.
  - `defend`.
  - `take_cards`.
  - `pass`.
- При входе в комнату:
  - проверка sessionId.
  - проверка прав/идентичности игрока.
  - синхронизация игрока в Redis.
- После каждого действия:
  - получить state из Redis.
  - применить действие через `GameEngine`.
  - сохранить новый state в Redis.
  - разослать персонализированный `state_updated`.
  - опубликовать событие в `game-events`.
- При завершении:
  - отправить `game_over`.
  - зафиксировать статус сессии.

### Frontend

- Реализовать `SocketService`:
  - connect/disconnect.
  - emit.
  - обработка входящих событий.
- Реализовать игровое состояние через Signal API.
- Обновлять UI строго событиями сервера (сервер — источник истины).
- Добавить recovery-сценарии:
  - reconnect.
  - повторная синхронизация состояния после reconnect.

### Критерий готовности

- Два игрока играют полный матч в real-time без рассинхронизации состояния.
- Переподключение не ломает игру.

## 12) Приоритеты реализации (что делать первым)

При конфликте задач придерживаемся такого порядка:

1. Контракты в `shared` (иначе будет дублирование и конфликт типов).
2. Auth + Sessions (иначе нельзя стабильно открыть матч между игроками).
3. DurakEngine с тестами (иначе невозможно доверять игровой логике).
4. GameGateway и синхронизация через Redis.
5. UI-полировка и анимации.

## 13) Definition of Done для MVP

MVP считается завершенным, если одновременно выполняется все ниже:

- Работает регистрация, логин, refresh, logout, guest flow.
- Игрок 1 создает сессию, игрок 2 подключается по ссылке.
- Полная партия "Дурака" проходит в real-time без потери состояния.
- Состояние игроков персонализировано (чужие карты скрыты).
- Основные события пишутся в Kafka и сохраняются в историю.
- Основные сценарии покрыты тестами и проходят в CI/локально.
- Есть документация по запуску dev/prod.

## 14) Риски и контрольные точки

### Риск 1: рассинхронизация состояния между клиентами

- Митигация:
  - сервер как source of truth.
  - персональная рассылка состояния после каждого действия.
  - отдельные тесты на reconnect/late join.

### Риск 2: усложнение логики "Дурака"

- Митигация:
  - разделение логики на чистые функции.
  - таблица правил + edge-case тесты до UI-этапа.

### Риск 3: деградация auth/WS безопасности

- Митигация:
  - единые guard-ы для HTTP/WS.
  - blacklist logout токенов в Redis.
  - rate limit на auth endpoints.

### Риск 4: дублирование логики между frontend/backend

- Митигация:
  - все контракты только в `shared`.
  - обязательный code review на предмет дублирования типов.

## 15) Что можно расширять после MVP

- Рейтинг и статистика игроков.
- История партий с replay-режимом.
- Новые игры через добавление новых engine.
- Матчмейкинг без ручной ссылки.
- Наблюдаемость и метрики (latency, disconnect rate, error rate).
